# HoopUp

Basketball pickup-game coordinator. Find a court, pick a time, get on the floor.

Live: https://mhayeri.github.io/hoopup/

## Stack

- **Frontend:** React 19 + Vite 6 + TypeScript 5.7 + Tailwind CSS 4.3
- **Routing:** React Router 7 (HashRouter for GitHub Pages)
- **Map:** Leaflet + React Leaflet + OpenStreetMap Overpass API
- **Backend:** Supabase (Postgres 15 + PostGIS + Auth + Storage + RLS)
- **Deploy:** GitHub Pages via GitHub Actions CI
- **Supabase project ID:** `cnqtnomndqcuxeogzihb`

## Database schema (public)

| Table             | Key columns                                                                                                   | Notes                                                                                                                                                                                                                                                                                                           |
| ----------------- | ------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `profiles`        | id (uuid, FK auth.users), username, skill_level, preferred_position, years_playing, home_court_id, avatar_url | Auto-created on signup via trigger. RLS: owner-only writes.                                                                                                                                                                                                                                                     |
| `courts`          | id (bigserial), osm_id, name, lat, lng, surface, hoops, lit, geom (PostGIS)                                   | Read-only for clients. Upserted via `upsert_osm_courts()` SECURITY DEFINER RPC.                                                                                                                                                                                                                                 |
| `sessions`        | id (uuid), court_id, host_id, starts_at, ends_at, notes, cancelled_at                                         | Host-only writes via RLS. DB triggers block post-start edits and un-cancellation.                                                                                                                                                                                                                               |
| `session_rsvps`   | session_id + user_id (composite PK), status ('going'\|'waitlist'\|'cancelled')                                | User-only writes via RLS. 15-player cap enforced atomically by `enforce_session_cap()` SECURITY DEFINER trigger with row-level locking.                                                                                                                                                                         |
| `friendships`     | requester_id + addressee_id (composite PK), status ('pending'\|'accepted'), accepted_at                       | Directional pair so we know who asked whom. Unordered-pair UNIQUE index blocks A‚ÜíB/B‚ÜíA coexistence. RLS: anon SELECT of accepted rows powers public friend lists; only addressee can flip pending‚Üíaccepted; either party can DELETE. Triggers: P0004/P0005/P0006 + accepted_at auto-stamp + party immutability. |
| `court_favorites` | user_id + court_id (composite PK), created_at                                                                 | Per-user saved courts (distinct from `profiles.home_court_id`, which is a single home base). RLS: owner-only SELECT/INSERT/DELETE, no UPDATE (toggle semantics ‚Äî a row exists iff saved). Both FKs `on delete cascade`. No trigger.                                                                             |
| `auth_throttle`   | ip + window_start (composite PK), attempts                                                                    | Per-IP fixed-window login-attempt counter for the `login` edge function. RLS enabled with **no policies** ‚Üí only the service role (which the function holds) can read/write. Written via the `record_login_attempt()` SECURITY DEFINER fn (service-role-only grant).                                            |

## Key patterns

- **Hand-maintained TS types** in `src/lib/database.types.ts` ‚Äî keep in sync with migrations.
- **RLS + triggers** ‚Äî RLS enforces ownership; triggers enforce business logic (session cap, edit guards).
- **Debounced Overpass sync** ‚Äî map pan/zoom fetches courts from Overpass, upserts via RPC, per-bbox cache prevents duplicate queries.
- **Sanitized client-to-server** ‚Äî Overpass data validated + truncated before RPC call.
- **Optimistic UI** ‚Äî useProfile applies updates optimistically, reverts on error.
- **PG error codes** ‚Äî `P0001` = SESSION_FULL, `P0002` = SESSION_NOT_AVAILABLE, `P0003` = USERNAME_GENERATION_FAILED, `P0004` = FRIENDSHIP_SELF, `P0005` = FRIENDSHIP_DUPLICATE (reverse pair exists), `P0006` = FRIENDSHIP_IMMUTABLE.
- **Centralized error mapping** ‚Äî `friendlyMessage()` in `src/lib/errors.ts` converts raw Supabase/Postgres errors to user-friendly strings. All error call sites use this instead of `error.message`.
- **SECURITY DEFINER triggers** ‚Äî `enforce_session_cap()`, `upsert_osm_courts()`, `add_host_to_session_rsvps()`, `cancel_session_if_host_alone()`, `friendships_before_insert()`, and `friendships_before_update()` run as function owner to bypass RLS for cross-table locks/writes. Always paired with `SET search_path = public`. **Trigger-only functions REVOKE EXECUTE from public/anon/authenticated** so they can't be called as `/rest/v1/rpc/<name>` endpoints (triggers still fire regardless of the grant); service-role-only helpers (`record_login_attempt`, `resolve_login_email`) GRANT EXECUTE only to `service_role`.
- **Security hardening / `/security-audit`** ‚Äî the four security pillars (rate limiting, production-grade auth, DB lockdown, no secret exposure) are verified by the `/security-audit` skill (`.claude/skills/security-audit/`), which runs the live Supabase advisors (`get_advisors`) plus RLS/grant/secret/auth-config/rate-limit checks and emits a severity-ranked report (diagnose-only; complements the `security-enforcer` agent). Concrete hardening lives in migrations `20260525120000_security_lockdown` (revoke trigger-fn RPC surface, pin search_paths, drop the avatars listing policy) and `20260525120100_drop_get_email_rpc` (drop the email-enumeration RPC ‚Äî apply to live only **after** the new login path is deployed). Auth config in `supabase/config.toml`: `password_requirements`, `secure_password_change`, `[auth.rate_limit]`. **Dashboard action:** enable leaked-password (HaveIBeenPwned) protection; `config.toml` auth/rate-limit changes apply to the hosted project via `supabase config push` or the Dashboard.
- **Rate-limited username login** ‚Äî `supabase/functions/login` (`verify_jwt = false`, callers are logged out) resolves a username‚Üíemail server-side (`resolve_login_email`, never returned to the client), performs the sign-in, and hands back only the session tokens (client calls `supabase.auth.setSession`). Unknown username and wrong password return the **same** generic 401 ‚Üí no account enumeration; attempts are throttled per IP (`record_login_attempt` + `auth_throttle`, returns 429 over the limit). `LoginPage` invokes it via `supabase.functions.invoke('login', ‚Ä¶)`; this replaced the public `get_email_by_username` RPC. Password rule is shared client-side in `src/lib/password.ts` (`validatePassword`/`PASSWORD_HINT`, mirrors the server requirement) and used by signup, change-password, and the reset flow.
- **Password reset** ‚Äî `/reset-password` (`ResetPasswordPage`) sends `resetPasswordForEmail` with a `redirectTo` carrying a `flow=recovery` marker (neutral "if an account exists" copy ‚Üí no enumeration); `AuthCallbackPage` detects recovery (via `flow=recovery`/`type=recovery`) and routes to `/update-password` (`UpdatePasswordPage`), which sets the new password in the recovery session and guards against a missing/expired link.
- **Host is always a player** ‚Äî DB triggers keep `sessions.host_id` in sync with `session_rsvps`. On session INSERT the host is auto-added with status='going'; if the host's RSVP transitions to 'cancelled' (or is deleted) and no other 'going' rows remain, the session auto-cancels.
- **Edge Functions for admin-only auth ops** ‚Äî `supabase/functions/delete-account` uses the service-role key to call `auth.admin.deleteUser()` (not callable from Postgres/RLS). JWT is verified at the gateway (`verify_jwt = true`) and re-resolved in-function. Storage avatars are removed before the cascade fires.
- **Reusable Tabs primitive** ‚Äî `src/components/Tabs.tsx` is an ARIA-correct tab strip (roles, `aria-selected`, Left/Right arrow nav, pill styling). Caller owns panel content. Profile page is the first consumer; pattern is ready for future tabbed surfaces.
- **Player search** ‚Äî navbar search icon (auth-only) opens `PlayerSearchOverlay` (full-screen on mobile, centered panel on desktop, modeled on `Modal`'s a11y). `useProfileSearch` runs a debounced (`useDebouncedValue`), self-excluding, capped (8) **prefix** query: `username LIKE 'q%'` on a lowercased query, backed by the `profiles_username_prefix_idx` (`text_pattern_ops`) btree ‚Äî relies on the existing `profiles_select_all` RLS, no RPC. Each result reuses the morphing `FriendActionButton`; the avatar + pills block is the shared `ProfileIdentity` (also used by `FriendRow`). LIKE wildcards in the query are escaped.
- **Responsive NavBar** ‚Äî the desktop link cluster is `hidden sm:flex`; below `sm` (640px) the bar is just the wordmark + a hamburger that toggles `MobileNavMenu` (`src/components/MobileNavMenu.tsx`). The menu is a dropdown that drops under the sticky header and dims the page below it. It **portals to `<body>` at `z-[2000]`** for the same two reasons as `PlayerSearchOverlay`: the header's `backdrop-blur` clamps `position: fixed` descendants, and on `/map` it must paint above Leaflet (z-index up to ~1000). The portal container is `pointer-events-none` with `pointer-events-auto` children, leaving the top strip over the header click-through so the hamburger (which morphs to ‚úï at the header's z-10) stays tappable to close. Player search folds into the menu as a "Find players" item (closes the menu, then opens `PlayerSearchOverlay`); reuses the Escape / body-scroll-lock / focus-restore effect shape, plus a `matchMedia('(min-width: 640px)')` listener that auto-closes on resize to desktop so the scroll lock can't get stuck. The panel sits at `top-14` (3.5rem, the same header height `MapPage` assumes in `h-[calc(100vh-3.5rem)]`). HTML mockup at `design-mockups/navbar-mobile.html`.
- **Court favorites** ‚Äî owner-private `court_favorites` table (composite PK `user_id + court_id`, both FKs `on delete cascade`); distinct from the single `profiles.home_court_id`. Toggle semantics ‚Üí no UPDATE policy. `useFavoriteCourt(courtId)` powers the optimistic ‚òÖ Save pill on the court detail page (reads auth itself; save is `upsert` with `ignoreDuplicates` so a double-tap is idempotent, revert on error). `useFavoriteCourts(userId)` loads the list (court row embedded via PostgREST) with an optimistic `remove`, surfaced as a self-only **Favorites** profile tab. Un-saving from the list row is gated behind `RemoveFavoriteCourtModal` (shared `Modal` shell, modeled on `RemoveFriendModal`) so a misclick can't silently drop a saved court; the court-detail ‚òÖ stays a one-tap toggle. No RPC ‚Äî relies on the owner-only RLS. HTML mockup at `design-mockups/court-favorites.html`.
- **Map session filters** ‚Äî client-side filtering of the already-fetched upcoming-sessions list in `SessionPanel` (time window / open-spots-only / host skill). No extra round trip; the only query change is adding `skill_level` to the host embed in `useUpcomingSessions`. Filter state is local `useState` in the panel (not lifted ‚Äî filters narrow the list only and never touch map markers, which are driven separately by `useCourtsInView`/`useActiveCourts`). The time-window filter (`isWithinTimeWindow` in `formatTime.ts`) applies to **upcoming only** ‚Äî live "Hooping" games stay visible regardless; open-spots (`goingCount < SESSION_CAP`) and skill apply to both. Filter row shows only on the Sessions tab; the tab badge reflects the filtered count; a "No sessions match your filters ¬∑ Clear filters" state is distinct from the genuinely-empty one. HTML mockup at `design-mockups/map-session-filters.html`.
- **Home page (auth-aware)** ‚Äî `routes/HomePage.tsx` branches on `useAuth()`: while `loading`, a skeleton (avoids a marketing‚Üílaunchpad flash); signed out, a marketing split hero (copy + CTAs + live `StatsStrip`) beside `GamesNearbyRail`; signed in, a personalized launchpad ‚Äî greeting (username capitalized for display, stored lowercase) + `NextGameCard` (first entry from `useUserActiveSessions`, flips to live "Hooping"/waitlist labels) beside the rail, with `QuickActions` tiles below. The Friends / Saved-courts tiles deep-link via `?tab=` ‚Äî `ProfilePage` reads it with `useSearchParams` and clamps self-only tabs (`favorites`/`settings`) on public `/u/:username` routes. All read-only over existing hooks (`useUpcomingSessions` fetched once and passed to both the rail and stats; no new backend). `GamesNearbyRail` is presentational (sessions passed in, same pattern as `SessionPanel`), splits live/upcoming via `getSessionStatus` on a `useNow` tick, and `excludeSessionId` drops the next-game card's session from the rail. The rail's empty state distinguishes a genuinely-empty board ("No games scheduled yet. Be the first to host one.") from the case where the rail is empty _only because_ the next-run game was excluded ("That's the only game on the board right now‚Ä¶") ‚Äî so it never claims nothing is scheduled while the next-run card is showing one. `HomeSessionCard` mirrors `SessionCard`'s look but is a plain `<Link>` (no map-select state, no inline friend button) so it's safe signed-out. `StatsStrip` (Hooping now / Games today / This week) is derived client-side and hidden when empty. "Games near you" is global upcoming-by-start-time; true distance (geolocation / `home_court_id`) and live tile counts are deferred. HTML mockup at `design-mockups/home-revamp.html`.
- **Web fonts loaded** ‚Äî `index.html` loads the display (`Bebas Neue`) + body (`Inter`) fonts declared in `index.css`'s `@theme` (`--font-display` / `--font-body`) via Google Fonts. Without the `<link>` both stacks fell back to `system-ui` app-wide; loading them is what gives `font-display` headings their condensed look everywhere (home hero, `SessionPanel` title, etc.).
- **Volt theme (dark, app-wide)** ‚Äî the app uses a **dark "Volt night-neon"** look (roadmap #34), now applied to every screen. Tokens in `index.css` `@theme`: Volt tokens (`--color-night`/`-2`/`-3` dark surfaces, `--color-bone` text, `--color-blue` = STRUCTURE [links, outlines, selection, secondary], `--color-volt` = ENERGY [live status, primary CTAs, key numbers]) are what components use; the legacy light tokens (`--color-court`/`hardwood`/`chalk`/`net`/`ink`) are still defined but **no longer referenced anywhere** (a future cleanup can remove them). **Conventions:** dark `night-2`/`white/[0.03]` cards with `border-white/10`; `bone` text (`/70`,`/55`,`/45` muted); primary CTA = `bg-[var(--color-volt)] text-[#0c1402]` + glow shadow (`.volt-glow` on display headings); secondary/outline = `border-[var(--color-blue)]/50` or `border-white/15`; inputs = `bg-white/[0.04] border-white/15` + blue focus ring; live/"Hooping" = volt pill/dot; red kept for errors/danger (`border-red-500/40 bg-red-500/10 text-red-300`). The `body` background stays light (legacy default), so each **full-page route paints its own** `min-h-[calc(100dvh-3.5rem)] bg-[var(--color-night)] text-[var(--color-bone)]` wrapper (mirror `HomePage`; home adds the `.volt-floods` glow). `NavBar` is a global dark header. Dark Leaflet styling (container/popup/controls/attribution) lives in `index.css` with `!important` so it beats `leaflet.css`. Roster skill tiers on dark: beginner=`#34d399`, intermediate=`#2b6fff`, advanced=`#a78bfa`, pro=`#fbbf24`, unset=`rgba(255,255,255,0.3)`.
- **Roster (box-score)** ‚Äî the session-detail roster is a basketball box-score: a hardwood column header (`# ¬∑ Player ¬∑ Pos ¬∑ Skill ¬∑ Yrs`) over `PlayerRow` cards. The shared grid template lives in `ROW_GRID` (exported from `PlayerRow`) so the header and rows align; each group is wrapped in an `overflow-x-auto` + `min-w-[460px]` so columns hold on narrow screens. The 15 confirmed (`going`) players are sorted host-first (stable, so others keep the hook's `created_at` order) then split by `FLOOR_SIZE` (=10, exported from `useSessionRsvps`): first 10 ‚Üí **On the floor** (#1‚Äì10), next 5 ‚Üí **Bench** (#11‚Äì15) ‚Äî both real spots, both numbered; the overflow `waitlist` (16+) stays a collapsible **Waitlist** with a dash for the number. Each row has a **skill-tier left rail** (beginner‚Üígreen, intermediate‚Üícourt, advanced‚Üíhardwood, pro‚Üígold, unset‚Üígrey) echoed by the Skill-column text color. Host is pinned first with a üëë Host chip + court-colored number; your own row is court-tinted with a "You" chip. **No hover card** ‚Äî `@username` is a `<Link to="/u/:username">` (PR #26 redirects `/u/<self>`‚Üí`/profile`); friends are managed on the profile page (the old `PlayerHoverCard` was the only roster `FriendActionButton` and is removed). HTML mockup at `design-mockups/roster-redesign.html`.

## Design workflow

- **HTML mockups before UI plans** ‚Äî when planning how a feature looks (layout, placement, new screens), produce a self-contained HTML mockup under `design-mockups/` first, rendered in the real app theme (color tokens, fonts, pill/button styles), so layout choices can be reviewed visually before any plan is finalized or code is written. One file per feature (e.g. `design-mockups/friend-search.html`); show the realistic states, not just empty shells.
